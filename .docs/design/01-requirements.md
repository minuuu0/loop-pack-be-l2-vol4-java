# 01. 요구사항 명세

> 근거 문서: `docs/week2/round2-scenario.md`(요구사항 원문), `docs/week2/round2-quests.md`(설계 범위)
> 설계 결정 과정: `docs/week2/qna.md`

## 1. 개요

브랜드의 상품을 둘러보고, 좋아요로 관심을 표현하고, 여러 상품을 한 번에 주문·결제하는 이커머스다. 이번 설계는 **상품·브랜드 조회 / 좋아요 / 주문·결제**의 대고객 흐름과, 이를 운영하는 **어드민**의 골격을 다룬다.

## 2. 범위

### 2.1 포함

| 도메인 | 기능 |
| --- | --- |
| 상품·브랜드 | 상품 목록 조회(브랜드 필터·정렬·페이징), 상품 상세 조회, 브랜드 조회 |
| 좋아요 | 좋아요 등록·취소(멱등), 내가 좋아요한 상품 목록 조회 |
| 주문·결제 | 주문 생성(재고 차감·상품 스냅샷), 주문 목록/상세 조회, 결제(포인트 차감·외부 연동) |
| 어드민 | 브랜드·상품·주문 관리 — **존재·권한·핵심 제약만** 명세(상세 흐름은 대고객 위주) |

### 2.2 명시적 제외 (경계)

| 제외 | 이유 |
| --- | --- |
| 회원가입 / 내 정보 조회 / 비밀번호 변경 | week1(volume-1) 산출물 재사용, 이번 주 재설계 안 함 |
| 주문 취소 / 환불 | 원문 요구사항에 없음 |
| 장바구니 | 원문에 장바구니 API가 없다. 주문은 `POST /orders`로 항목(items) 배열을 직접 받아 즉시 처리하는 **cart-less** 방식이며, 장바구니 담기·조회·수정의 영속화는 범위 밖이다 |
| 쿠폰 발급·사용 | 배경에는 등장하나 API 없음 → 향후 확장 |
| 포인트 충전 | 차감만 범위. 충전은 제외 |
| 동시성·느린 조회 등 심화 | 원문상 "모든 기능 개발 후 해결" → 1차 범위 밖. 멱등성만 정책으로 명시 |

## 3. 액터

| 액터 | 설명 | 식별 |
| --- | --- | --- |
| 방문자 | 비로그인. 상품·브랜드 조회만 가능 | 없음 |
| 사용자 | 로그인. 좋아요·주문·결제 | `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더 |
| 어드민 | 브랜드·상품·주문 관리 | `X-Loopers-Ldap` 헤더 |
| 외부 결제 시스템 | 결제 요청에 성공/실패를 반환하는 외부 시스템(취소·환불 API 없음) | — |

> 인증/인가는 구현 범위가 아니며 헤더로 사용자/어드민을 식별만 한다. 사용자는 타 유저의 정보에 직접 접근할 수 없다.

## 4. 기능 요구사항

### 4.1 고객 — 상품·브랜드 탐색

고객은 브랜드와 상품 정보를 자유롭게 탐색하여 구매 의사결정을 내린다.

**시나리오**
1. 고객이 특정 브랜드의 정보를 조회한다.
2. 고객이 상품 목록을 조회한다. 브랜드(`brandId`)로 필터링하거나, `latest`(필수) / `price_asc` / `likes_desc`(선택) 기준으로 정렬하고, 페이지 단위로 조회한다.
3. 고객이 특정 상품의 상세 정보를 조회한다. 상세에는 좋아요 수가 함께 표시되며, 로그인한 고객에게는 본인의 좋아요 여부도 표시된다.

**전제**
- 상품·브랜드 탐색은 로그인 없이 가능하다(방문자 액터).

**예외**
- 존재하지 않거나 삭제된 상품·브랜드는 조회되지 않는다.

### 4.2 고객 — 상품 좋아요

고객은 관심 있는 상품에 좋아요를 표시하고, 좋아요한 상품을 다시 찾는다.

**시나리오**
1. 고객이 상품에 좋아요를 등록한다.
2. 고객이 좋아요를 취소한다.
3. 고객이 자신이 좋아요한 상품 목록을 조회한다.

**전제**
- 좋아요 등록·취소·목록 조회는 로그인이 필요하다(사용자 액터). 좋아요의 기준 키가 `(고객, 상품)`이므로 고객 식별이 전제다.
- 같은 고객이 같은 상품에 좋아요를 여러 번 요청해도 결과 상태는 **0 또는 1**이다(멱등).

**예외**
- 비로그인 요청, 존재하지 않는 상품에 대한 요청은 거부된다.

### 4.3 고객 — 주문·결제

고객은 여러 상품을 한 번에 주문하고 결제한다.

**시나리오** (주문 요청 시)
1. 고객이 주문 항목(상품, 수량)을 담아 주문을 요청한다.
2. 시스템이 상품 존재·판매 가능 여부를 확인한다.
3. 시스템이 재고를 검증하고 차감한다.
4. 시스템이 포인트를 검증하고 차감한다.
5. 시스템이 외부 결제를 연동한다(마지막, 비가역).
6. 주문을 확정하고, 주문 시점의 상품 정보(이름·가격 등)를 스냅샷으로 저장한다.
7. 고객이 자신의 주문 목록(기간별)과 단일 주문 상세를 조회한다.

**전제**
- 주문·결제·주문 조회는 로그인이 필요하다(사용자 액터).
- 주문은 장바구니 없이 `POST /orders`로 주문 항목(items)을 직접 받아 즉시 처리한다(cart-less). 장바구니 영속화는 범위 밖이다(§2.2).
- 차감 순서는 재고 → 포인트 → 외부 결제다. 재고·포인트는 우리 DB라 롤백이 싸고, 외부 결제는 비가역이므로 마지막에 두어 "결제 성공 후 우리 쪽 실패 → 환불 보상" 케이스를 회피한다.

**예외**

| 실패 지점 | 처리 | 고객에게 |
| --- | --- | --- |
| 재고 부족 | 주문 거부, 차감 없음 | "재고 부족" — 단계 식별 가능 |
| 포인트 부족 | 주문 거부, 재고 차감 롤백 | "포인트 부족" — 단계 식별 가능 |
| 외부 결제 실패 | 우리 DB 트랜잭션 롤백(재고·포인트 원복), 별도 보상 API 불필요 | "결제 실패, 다시 시도" — 상세 비노출(일반화) |

### 4.4 어드민 — 브랜드·상품·주문 관리

어드민은 카탈로그와 주문을 운영한다. (존재·권한·핵심 제약만 명세, 상세 흐름은 대고객 위주)

**시나리오**
1. 어드민이 브랜드를 등록·수정·삭제하고, 브랜드 목록·상세를 조회한다.
2. 어드민이 상품을 등록·수정·삭제하고, 상품 목록·상세를 조회한다.
3. 어드민이 주문 목록·상세를 조회한다.

**전제**
- 어드민 기능은 `X-Loopers-Ldap` 헤더로 어드민을 식별한다(어드민 액터). 대고객 prefix(`/api/v1`)와 어드민 prefix(`/api-admin/v1`)는 분리된다.

**핵심 제약**
- 브랜드 삭제 시 해당 브랜드의 상품들도 함께 삭제된다.
- 상품 등록 시 브랜드는 이미 등록된 브랜드여야 한다.
- 상품 수정 시 브랜드는 변경할 수 없다.
- 상품·브랜드 정보 중 고객 노출 항목과 어드민 전용 항목을 구분한다(상세는 클래스/ERD 단계에서 확정).

## 5. 정책 & 제약사항

- **좋아요 멱등성**: `(user, product)`당 좋아요는 0 또는 1. 중복 요청에도 결과 동일.
- **주문 차감 순서**: 재고 → 포인트 → 외부 결제. 외부 결제 실패 시 우리 DB 롤백으로 일관성 보장(보상 불필요).
- **주문 스냅샷**: 주문 후 상품 정보가 바뀌어도 주문에는 주문 시점 정보가 보존된다.
- **재고**: 재고는 음수가 될 수 없다(차감 전 검증).
- **포인트**: 잔액 미만 결제 불가. 충전은 범위 밖.
- **접근 제어**: 사용자는 타 유저 정보에 직접 접근 불가. 어드민/대고객 prefix(`/api-admin/v1`, `/api/v1`)로 분리.

## 6. 가정

- 요구사항 원문(`round2-scenario.md`)에 명시되지 않은 부분은 아래 전제로 진행한다.
- 인증/인가는 구현하지 않고 헤더로 식별만 한다. 회원·인증 도메인은 week1 산출물을 재사용한다.
- 외부 결제 시스템은 "결제 요청 → 성공/실패 반환"만 하는 추상 시스템으로 가정한다(취소·환불 API 없음). 이 가정이 4.3의 차감 순서를 정당화한다.
- 동시성·멱등성의 구현 메커니즘(유니크 제약, 락 등)은 ERD/클래스 설계 단계에서 다룬다.
